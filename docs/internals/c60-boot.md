# C60 boot chain

The Trio C60 (codename `kepler_proto1`, i.MX 8M Mini) is the TC8's
sibling with the opposite security posture: **HAB is open**, so the
BootROM runs unsigned bootloaders. That removes the whole
chainload apparatus the [TC8 needs](tc8-boot.md) — the C60 gets a
replacement U-Boot loaded directly, and boots Linux with plain `booti`.

## HAB-open: SDP RAM-load

With HAB open, the BootROM's **SDP** recovery mode (USB HID device
`1fc9:0134`) accepts and runs an unsigned bootloader. Loading is
two-stage, the standard i.MX 8M Mini sequence:

1. BootROM SDP loads the SPL into on-chip RAM and jumps to it.
2. The SPL brings up DDR and re-enumerates as **SDPV**, which loads the
   full U-Boot FIT into DRAM and jumps.
3. The running U-Boot presents a fastboot gadget (`1fc9:0152`).

The provisioning wizard drives this entirely in the browser: the
BootROM's vendor HID interface is reachable over **WebHID**, so no
`uuu` binary and no driver install is needed
([provisioner architecture](provisioner.md)). Once the open U-Boot is
running, it is persisted into the eMMC boot area through the built-in
fastboot `bootloader0` target, and the rest of the install is the same
fastboot flow as the TC8.

## Boot model: `booti`

The C60 boots Android-format images without the Android machinery:

- **`boot.img`** — Android boot.img v0 carrying the kernel with the
  **DTB in the `second` area**. No ramdisk: the kernel mounts the
  rootfs directly, read-write — there is no
  [sealed-overlay initramfs](ro-root.md) on the C60.
- **`dtbo.img`** — the device tree in an Android DTBO container.
- **`vbmeta.img`** — AVB metadata chaining both, **signed**
  (`SHA256_RSA2048`). With HAB open the signature is not fuse-verified,
  but the stock U-Boot requires a structurally signed vbmeta carrying
  the dtbo descriptor, so the build signs rather than shipping
  `--algorithm NONE` as the TC8 does.

U-Boot reads the slot images out of eMMC, places kernel and DTB, sets
`bootargs`, and jumps with `booti`. The load/copy addresses depend on
the packed kernel size, which is why they travel in the release
manifest (below) rather than being hardcoded anywhere.

## Rootfs: `system_a`

The rootfs lives in the stock `system_a` partition
(`root=PARTLABEL=system_a`), mounted read-write like ordinary Linux.

`system_a` is **1.75 GiB** — 1,879,048,192 bytes, 3,670,016 × 512-byte
sectors — and that is a hard ceiling: the builder refuses to emit an
image that would not fit rather than truncate it. The default image
size is a headroom-safe 1.6 GiB.

Two rootfs artifacts exist, one per audience:

| Artifact | Producer | Role |
|---|---|---|
| `rootfs.simg` | release | Android **sparse** ext4, flashed to `system_a` over fastboot; named by the release's `c60-manifest.json`. |
| `rootfs.img.zst` | `build.sh --target=c60` | zstd-compressed raw ext4 — the development-build output. |

## Manifest-driven releases

Where TC8 releases rely on well-known asset names, a C60 release
carries a real **`c60-manifest.json`** asset, generated by the build.
It has to: the `bootSeq` it contains — the `mmc read`/`cp.b`/`booti`
sequence with its build-specific addresses — changes with every image
build. The manifest names the `flash.bin` bootloader (with size and
sha256), the boot sequence, and an optional `os` section listing the
OS images and their target partitions; when `os` is absent the wizard
offers unlock only. Its presence as a release asset is what makes a
release appear in the wizard's C60 OS chooser.

The manifest schema is owned by the wizard side and documented in
[`C60.md` in the provisioner repo](https://github.com/Polycom-Open-Firmware/provisioner/blob/main/C60.md);
the delivery pipeline is covered in [Releases and delivery](releases.md).
